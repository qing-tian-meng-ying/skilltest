# 中文 Emoji 提交信息生成器

一个用于生成符合中文团队规范的 Git 提交信息的 skill。

## 何时使用

- 用户在 git 提交前征求 commit message 建议时
- 用户问"这次改动应该怎么写提交信息"时
- 用户明确要求生成 commit message 时
- 看到 `git diff` 上下文并被请求总结改动时

## 提交信息格式

```
<类型>: <emoji> <简短中文描述>

<可选的详细说明，每行不超过 72 字符>
```

### 类型枚举

| 类型 | Emoji | 含义 |
|------|-------|------|
| feat | ✨ | 新功能 |
| fix | 🐛 | 修复 bug |
| docs | 📝 | 文档变更 |
| style | 💄 | 代码格式（不影响功能） |
| refactor | ♻️ | 重构 |
| perf | ⚡ | 性能优化 |
| test | ✅ | 测试相关 |
| chore | 🔧 | 构建/工具链/杂项 |

## 执行步骤

1. **收集上下文**：先查看 `git status` 和 `git diff --staged`，理解本次改动
2. **判断类型**：根据改动性质从上面的表格中选定 type
3. **撰写主标题**：
   - 总长度控制在 50 字符内
   - 使用动词开头：`添加`、`修复`、`重构`、`优化`、`移除`
   - 不加句号
4. **可选展开**：如果改动复杂（多文件、多模块），追加正文，分点列出关键变更
5. **输出验证**：检查是否符合 `<类型>: <emoji> <描述>` 格式

## 输出示例

简单改动：
```
feat: ✨ 添加用户登录接口
```

复杂改动：
```
refactor: ♻️ 重构数据处理模块的错误处理逻辑

- 提取统一的异常基类 DataProcessError
- 移除散落在各处的 try/except 重复代码
- 为关键路径添加 traceback 日志
```

## 反模式（不要这样做）

- ❌ 英文描述：`feat: add login api`
- ❌ 缺少 emoji：`feat: 添加登录接口`
- ❌ 标题超过 50 字：长描述放正文
- ❌ 含糊不清：`fix: 🐛 修复 bug`（要说清楚修了什么 bug）
